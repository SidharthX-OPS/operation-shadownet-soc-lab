# ALERT-CRITICAL-T1070-Log-Cleared

| Field | Value |
|---|---|
| Rule ID | DET-005 |
| Severity | CRITICAL |
| MITRE Technique | [T1070.001 — Indicator Removal: Clear Windows Event Logs](https://attack.mitre.org/techniques/T1070/001/) |
| Data Source | Windows Security EID 1102, System EID 104 |
| Index | wineventlog |
| Validation Status | PASS |

---

## Description

Detects clearing of the Windows event logs. EID 1102 is generated by the Security log clearing operation itself; EID 104 covers System and Application log clears. Both events survive the clear by design — they are written *as part of* the clearing operation, so they always exist after the fact.

This is the rule that catches attackers covering their tracks. There is no legitimate reason for a non-administrator to clear the Security log on an active system, and very few legitimate reasons even for administrators outside of incident response or scheduled compliance retention.

## SPL

```spl
index=wineventlog (EventCode=1102 OR EventCode=104) earliest=-30m
| eval severity="CRITICAL",
       action="Log Cleared — Possible Attacker Covering Tracks"
| table _time, ComputerName, source, EventCode, action, severity
| sort - _time
```

## Validation Evidence

Triggered against `wevtutil cl Security`, `wevtutil cl System`, `wevtutil cl Application`. Three events captured in Splunk within 30 seconds of execution. All three forwarded to the SIEM before the local clear took effect — off-host log retention is the architectural control that makes this detection possible at all.

## Critical Architecture Note

This rule is only useful if logs are forwarded off-host **before** being cleared. If the SIEM only ingests logs at a polling interval (or worse, only when queried), an attacker can clear logs faster than they're shipped. The Splunk Universal Forwarder ships in near-real-time (sub-second under normal conditions), which is what allows this detection to work.

In production deployments, this assumption must be verified — not assumed.

## False Positive Notes

Legitimate scenarios:
- IR teams clearing logs after an incident as part of cleanup (rare; usually preserved instead)
- Compliance teams running scheduled log retention scripts (these should be documented and whitelisted)
- IT operations during major maintenance windows

**Tuning approach:** Do not tune this rule down. Every EID 1102 should be investigated. Whitelisting is appropriate only for documented, scheduled, approved log management automation — and even then, the whitelist should require both the user account *and* the time window.
